## What worked

- Deep link reaches the app reliably from simulator URL open.
- Signed-out gate clearly communicates required auth with "Sign in required".
- Onboarding CTA includes "I have an account", which is the right route for an invited first-timer.

## Friction (prioritized)

| P | Screen | Issue | Suggested change | Contract impact |
|---|--------|-------|------------------|-----------------|
| P0 | Invite auth gate on onboarding | Deep link lands on onboarding + modal, but progression into invite flow is not deterministic; repeated modal interactions are required and first-timer intent is easy to lose. | Introduce a dedicated "pending invite" state after auth-required modal: persist invite payload + route automatically to invite target after login. | `flow-invite-to-rally` should explicitly require pending deep-link recovery after auth. |
| P1 | Onboarding action hierarchy | "I have an account" is present but competes with carousel context; first-timer can miss that this is the next step for invites. | Add contextual subcopy when deep-link originated: "You were invited to a Rally. Sign in to continue." | `flow-invite-to-rally` copy/UX section should include signed-out invite text requirements. |
| P2 | Language clarity | Carousel still mixes "Rally" and "games"; for a first-timer invite path this is acceptable but not ideal. | Keep invite pathway language consistently "join this game/Rally" on the auth gate. | Minor copy alignment note in `flow-rally-session`. |

## Recommended contract changes

- [ ] Add a mandatory "deep-link payload persists through auth gate and resumes target route" criterion to `flow-invite-to-rally`.
- [ ] Add signed-out invite copy requirements for first-time users.
