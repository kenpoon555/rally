### Finding 1 тАФ 118 `as never` nav casts: root cause is fragmented ParamList (STRUCTURAL тЖТ ADR)

Every `navigate()` call that crosses a navigator boundary without a shared type falls back to `as never`:

```ts
// GameRoomActionBar.tsx:1373
navigation.navigate('MainTabs' as never, { screen: ... } as never);

// HomeScreen.tsx:428
navigation.getParent()?.navigate(ROUTES.ACTIVITY.DETAIL as never, { activityId } as never);

// ChatThreadScreen.tsx:407
navigation.replace(ROUTES.REGULAR_GROUP.CREW as never, { groupId, initialTab } as never);
```

**Root cause:** `RootStackParamList` in `navigationRef.ts` only covers 6 routes out of 22 registered in `AppNavigator.tsx`. There is no global type augmentation for React Navigation's `RootParamList`, so the type system can't resolve routes from nested navigators.

**What typed navigation looks like with React Navigation:**
```ts
// src/navigation/types.ts (new file)
export type RootStackParamList = {
  MainTabs: NavigatorScreenParams<MainTabParamList>;
  [ROUTES.ACTIVITY.DETAIL]: { activityId?: string; inviteToken?: string; ... };
  [ROUTES.ACTIVITY.CREATE]: { createMode?: 'sport' | 'class' } | undefined;
  [ROUTES.CHAT.THREAD]: { conversationId: string; title?: string; activityId?: string };
  [ROUTES.REGULAR_GROUP.CREW]: { groupId: string; initialTab?: string };
  // ... all 22 routes
};

// React Navigation global augmentation (eliminates as never everywhere):
declare global {
  namespace ReactNavigation {
    interface RootParamList extends RootStackParamList {}
  }
}
```

With this in place, `navigation.navigate(ROUTES.ACTIVITY.DETAIL, { activityId })` becomes valid TypeScript тАФ no cast needed.

**Baseline:** 118 casts. After ADR: 0 casts.

| `FeedbackScreen.tsx` | 9тАУ12 | `Feedback` only |

Each definition is a different subset. If `ChatThread`'s params gain a new required field:
- `ActivityDetailScreen.tsx` still compiles (it defines its own `ChatThread: { conversationId }`)
- The new field is silently missing at the call site
- TypeScript does not catch it because the local type matches the local definition

With a single shared `RootStackParamList`, this class of bug disappears.

### Finding 3 тАФ Navigation prop drilling at 2 sub-component sites

`RallyChatPanel.tsx:46` and `RallyPlayPanel.tsx:33` accept `navigation` as a prop (typed `NativeStackNavigationProp<any>`) drilled from `RegularsCrewScreen`. Both components could call `useNavigation<NativeStackNavigationProp<RootStackParamList>>()` internally тАФ they are deeply mounted components with a stable navigator ancestor, not headless utilities.

Prop-drilling navigation is an anti-pattern here because:
1. It locks the component to be rendered inside a screen (not reusable standalone)
2. The prop is `<any>` тАФ no type safety
3. Every navigate call in the sub-component then requires `as never`

## Structural decision тЖТ ADR

**ADR-0003 (new): Typed navigation тАФ single `RootStackParamList` + global type augmentation.**

Scope:
1. Create `src/navigation/types.ts` тАФ single source of truth for all stack, tab, and auth param lists
2. Augment `ReactNavigation.RootParamList` тЖТ eliminates `as never` at call sites
3. Remove 6 local `*ParamList` definitions (each file imports from `types.ts`)
4. Switch `useNavigation<any>()` тЖТ `useNavigation<NativeStackNavigationProp<RootStackParamList>>()`
5. Remove `navigation` props from `RallyChatPanel` and `RallyPlayPanel`, use `useNavigation()` internally

**Effort:** ~4тАУ6h. Medium LOC change (mostly `import` and cast removal), zero behaviour change. Can land without any feature work; 0 risk of regression.

**Why defer vs do now:** Low urgency тАФ the `as never` casts work at runtime; this is a future-proofing fix. High value for the reactions PR and `useChatChannel` refactor, because those both add new routes or navigate from within hooks/components. Draft ADR-0003, queue for next builder session.
