### Finding 1 — `withRetry`: immediate single retry, no backoff

`chatService.ts:8–21`:
```ts
const withRetry = async <T>(action: () => Promise<T>, retries = 1): Promise<T> => {
  let lastError: unknown;
  for (let attempt = 0; attempt <= retries; attempt++) {
    try {
      return await action();
    } catch (error) {
      lastError = error;
      if (attempt === retries) throw error;
    }
  }
  throw lastError;
};
```

Retry fires immediately (0ms gap). This is used for `getConversationMessages`, `sendConversationMessage`, `getConversationMembers`, `getLastMessagePreviews`, and `markConversationRead`.

**Today:** Supabase is reliable; the extra attempt is a negligible cost and saves the user from a transient 500.  
**Wall:** if Supabase is returning 503 (rate limit or maintenance) the retry doubles the load on an already-stressed endpoint. For send-message specifically, a double-send on a non-idempotent write is a risk (though the current implementation sends once and retries only the insert — if the insert succeeded the first time and returned a 500 response, the retry inserts a duplicate).

**Mechanical fix:** add a 300ms delay between retries for write operations. For reads, immediate retry is fine. Or: add a `delayMs` param to `withRetry` and pass `300` for send paths.

**Duplicate-send risk assessment:** Low today. `sendConversationMessage` currently has no idempotency key (no client-generated UUID per message). If a 500 response comes back after a successful DB insert, the retry creates a duplicate message. This is the higher-priority fix in `sendConversationMessage` — add a client-generated `id` field and make the insert `ON CONFLICT DO NOTHING`.

### Finding 2 — Realtime reconnect: no backfill (P1)

`chatService.ts:430–460` `subscribeToConversationMessages`:
```ts
supabase.channel(`conversation-${conversationId}`)
  .on('postgres_changes', { event: 'INSERT', table: 'messages', ... }, cb)
  .subscribe();
```

Supabase JS v2 Realtime reconnects automatically after a network drop. But the reconnect only re-establishes the subscription — it does **not** backfill messages sent during the outage. The `cb` only fires for new `INSERT` events after the reconnect moment.

**Scenario:** user on a spotty connection in a 20-person game chat. Network drops for 30 seconds; 4 messages are sent by others. Network restores, Realtime reconnects. User sees no indication of missed messages and the 4 messages never appear.

**Only recovery path:** navigate away and back (which calls `loadMessages` again). If the user stays on the screen, they're silently missing messages.

**Mechanical fix:** listen to the channel's subscription status callback for reconnect events and re-fetch:
```ts
channel.subscribe((status) => {
  if (status === 'SUBSCRIBED' && hasLoadedOnce) {
    void reloadRecentMessages(); // fetch last 20 messages and merge
  }
});
```

This is also a prerequisite for ADR-0002's `useChatChannel` hook — the reconnect backfill should live in the hook's subscription lifecycle, not in the component.

### Finding 3 — `loadJoinRequests` error: silent in host flow

`GameRoomActionBar.tsx:290–300`:
```ts
const loadJoinRequests = useCallback(async () => {
  setLoadingRequests(true);
  try {
    const requests = await getActivityJoinRequests(activityId);
    setJoinRequests(requests);
  } catch (error) {
    console.error('Error loading join requests:', error);
    // ← no Alert, no errorText in UI
  } finally {
    setLoadingRequests(false);
  }
}, [activityId]);
```

A host reviewing join requests gets the "Review" tab, which renders `joinRequests` state. If `loadJoinRequests` fails (transient network error, RLS change, session expiry):
- `joinRequests` remains at its previous value (or empty array from initial state)
- No feedback to the host — the panel either shows nothing or shows stale data
- Host might approve/reject based on stale data, or silently skip the panel thinking no requests exist

**Mechanical fix:** set a local `requestsError` state and render a "Could not load requests. Pull to refresh." banner in the host panel.

const previous = statusFor(row);
setLocalStatus((current) => ({ ...current, [row.id]: status })); // optimistic
setBusyId(row.id);
try {

This is the correct pattern: snapshot the previous value, apply optimistically, rollback on catch, surface the error to user. No action needed.

### Finding 6 — Bootstrap / stale session recovery (CORRECT)

`AuthContext.tsx` bootstrap path:
- `BOOTSTRAP_GET_SESSION_MS = 8s` — if `supabase.auth.getSession()` hangs, throw
- `BOOTSTRAP_LOAD_USER_MS = 10s` — if profile load hangs, call `clearStaleBootstrapSession()`
- `BOOTSTRAP_LOADING_BACKSTOP_MS = 12s` — safety `setTimeout(() => setLoading(false))` so the app never stays on a loading screen indefinitely
- `clearStaleBootstrapSession`: `signOut({ scope: 'local' })` (local only, preserves server session) → `setUser(null)` → AuthContext routes to Welcome

Three independent guards means the app recovers even if two of them malfunction. This is robust. No action needed.
