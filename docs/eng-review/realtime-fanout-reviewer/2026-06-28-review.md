### Finding 1 — reactions prerequisite: no channel-sharing model (STRUCTURAL → ADR)

`subscribeToConversationMessages` (`chatService.ts:430`):
```
supabase.channel(`conversation-${conversationId}`)
  .on('postgres_changes', { table: 'messages', ... }, cb)
  .subscribe()   // ← channel is subscribed immediately, returned to caller
```

The function creates and **immediately subscribes** a channel, then returns it to the caller purely for teardown (`subscription.unsubscribe()`). Once subscribed, Supabase JS v2 does not allow adding more `.on()` listeners.

When reactions ship (P1), they need `message_reactions` INSERT events on the same conversation. The only options today:
- A second call to this function → opens a **2nd `channel('conversation-{id}')`** in Supabase's channel pool (2 sockets per open chat)
- A new `channel('reactions-{id}')` → also 2 sockets per chat, different name
- Neither is acceptable at scale (200-member rally = 200 clients × 2 channels = 400 concurrent connections vs 200)

**Required:** a channel-ownership model where one hook/context holds the channel open for the screen's lifetime, registers all listeners before subscribing, and tears down on unmount. → **ADR-0002**

### Finding 2 — async race: host channels can leak on fast unmount

`useJoinRequestNotifications.ts:29–62` (`setupHostListeners`):
```ts
const setupHostListeners = async () => {
  const { data: hosted } = await supabase.from('activities')...  // ← await
  for (const activityId of hosted) {
    const channel = supabase.channel(`host-join-${activityId}`).subscribe();
    channels.push(channel);   // pushed AFTER the await
  }
};
void setupHostListeners();   // fire-and-forget

return () => {
  for (const ch of channels) { supabase.removeChannel(ch); }  // runs synchronously on unmount
};
```

If the component unmounts in the ~50ms window while the Supabase query is in-flight:
- Cleanup runs with `channels = []` (nothing to remove)
- `setupHostListeners` resolves later, subscribes and pushes channels after cleanup already ran
- Those channels are never removed → **one leaked socket per hosted activity** per fast-unmount cycle

Probability is low (logout/remount flow, not normal nav), but real. Each hosted game = one leaked socket.

**Mechanical fix:** use an abort flag pattern:
```ts
let cancelled = false;
const setupHostListeners = async () => {
  const { data: hosted } = await supabase.from(...)...
  if (cancelled) return;
  for (const activityId of hosted) { ... channels.push(...) }
};
return () => { cancelled = true; channels.forEach(ch => supabase.removeChannel(ch)); };
```

### Finding 3 — duplicate `channel('activities')` across mounted tabs

`useActivities.ts:78`:
```ts
supabase.channel('activities').on('postgres_changes', { table: 'activities' }, ...).subscribe()
```

`HomeScreen.tsx:279` calls `useActivities(location)` → opens `channel('activities')` #1.
`MapScreen.tsx:44` calls `useActivities(location)` → opens `channel('activities')` #2.

React Navigation bottom tabs keep both screens mounted once visited. After the user opens both tabs:
- Every activities change fires **2 independent `fetchActivities`** callbacks (one per tab)
- `useMyGames` (also on HomeScreen) opens a third channel: `channel('reload-activities,join_requests')` via `useSupabaseRealtimeReload`
- Total activities-watching channels on HomeScreen + MapScreen: **3**

At current scale (42 activities, infrequent writes): imperceptible. At 5k activities with match finalizations running frequently: 2 parallel discover re-fetches per event, wasting bandwidth.

**Mechanical fix:** scope the channel name to the screen/user: `channel('activities-discover')` vs `channel('activities-map')` to prevent name collisions and clarify intent. Or deduplicate via a shared singleton hook.

### Finding 4 — `getLastMessagePreviews` fallback: unbounded row scan

`chatService.ts:477–499`:
```ts
// fallback when get_inbox_message_previews RPC fails
const { data } = await supabase
  .from('messages')
  .select('conversation_id, content, created_at')
  .in('conversation_id', conversationIds)
  .is('deleted_at', null)
  .order('created_at', { ascending: false })
  // ← no .limit()
```

Then dedupes to one row per conversation in JS. At 100 conversations × 200 messages = 20k rows pulled, all deduped client-side.

**Today:** The RPC `get_inbox_message_previews` (migration `031`) is live and healthy; this fallback is dead code path. Risk is if the RPC errors (Supabase outage, schema change) and the fallback fires at scale.

**Mechanical fix:** add `.limit(conversationIds.length)` to the fallback query — this caps it at N rows (one per conversation in the best case given the `ORDER BY created_at DESC`). Or: cap the fallback entirely at 20 conversations and log a warning if exceeded.
