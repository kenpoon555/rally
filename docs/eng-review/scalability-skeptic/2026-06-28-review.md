### Finding 1 — Discover: no server-side geo-filter (CRITICAL path to ADR-0001)

`activityService.ts:267` `queryDiscoverActivities`:
```ts
supabase.from('activities')
  .select(DISCOVER_SELECT)
  .eq('status', 'active')
  .order('start_time', { ascending: true })
  .limit(120)   // ← global cap, no WHERE ST_DWithin(...)
```

DB returns the **120 soonest-starting active games globally**, then `getNearbyActivities` filters client-side:
```ts
const filtered = enriched.filter((activity) => {
  return activityWithinRadius(activity, latitude, longitude, effectiveRadius)
  // ...
});
```

**The wall:** when ≥120 active games exist outside the user's 60km radius (e.g., NYC has 500 active games), the LIMIT fills with those remote games before any local game is returned. SF user's discover feed → 0 results. This is silently wrong: no error, no empty-state explanation, just a blank feed.

**Today (42 activities globally):** safe — 120 cap never binds and all pass client-side filter.  
**Wall:** ~500 active games in any single metro.

**This is exactly ADR-0001's scope.** The accepted decision builds `discover_activities(p_lat, p_lng, p_radius_m, ...)` as a PostGIS server-side RPC. No further structural work needed here beyond ADR-0001 execution.

### Finding 2 — Chat thread: 100-message hard cap, no scroll-back pagination

`ChatThreadScreen.tsx:449–465`:
```ts
const loadMessages = useCallback(async () => {
  const rows = cached ?? (await getConversationMessages(conversationId, 100));
  setMessages(rows);
  // ...
```

`getConversationMessages` (`chatService.ts:312`) fetches the 100 most recent messages, reversed for display. The `FlatList` (`ChatThreadScreen.tsx:154`) renders those 100 with no `onEndReached` callback.

**The wall:** any group chat that accumulates >100 messages permanently hides history from the user. No "load older messages" affordance exists anywhere in the component tree.

**Today:** Demo thread has <50 messages. Safe.  
**Wall:** a 200-person rally active for a few days easily hits 200+ messages. The moment message #101 is sent, message #1 is gone from every client.

**Mechanical fix:**
1. Add keyset `before` cursor to `getConversationMessages`: `.lt('created_at', cursor).limit(50)`.
2. Wire `FlatList.onEndReached` to load earlier pages and prepend.
3. Move hard-coded `100` to a named constant `CHAT_PAGE_SIZE = 50`.

### Finding 3 — Discover enrichment: unbounded approved join_requests scan

`activityService.ts:235–239`:
```ts
supabase
  .from('join_requests')
  .select(JOIN_REQUEST_CARD_SELECT)
  .in('activity_id', activityIds)   // up to 120 IDs
  .eq('status', 'approved')         // no .limit()
```

For 120 discover cards, every approved joiner row is fetched. Each row includes `profiles` via nested join.  
**Row math:** 120 activities × 20 average approved joiners = 2,400 rows today; at 120 activities × 100 joiners = 12,000 rows.

**Practical cap today:** current typical roster size is 4–12. Safe.  
**Wall:** if Rally games scale to 50-player pickup events (tournament brackets, large park runs), this fetch can hit 5k+ rows per discover refresh.

**Mechanical fix:** add `.limit(activityIds.length * 20)` — caps at 2,400 rows (20 joiners × 120 activities), sufficient to show the "who's coming" preview without fetching full 200-person rosters. Display layers already cap preview rendering; the extra rows are never shown.

### Finding 4 — Inbox: no limit on `getMyConversations`

`chatService.ts:242–273`:
```ts
supabase
  .from('conversation_members')
  .select(`conversation:conversations(*, activity:activities(...))`)
  .eq('user_id', userId)
  .eq('is_active', true)
  // no .limit()
```

Returns every active conversation for the user, then JS sorts by `updated_at`. At 300 conversations this is 300 rows + 300 joined activity rows = 600 DB rows pulled on every inbox load + realtime refresh.

**Today:** active users might have 10–20 conversations. Safe.  
**Wall:** a power user on 50 games/month for a year accumulates 600+ conversations. Inbox load → 1,200 rows per open.

**Mechanical fix:** add `.limit(200).order('updated_at', { ascending: false })` — pushing the sort to DB, capping at 200 rows. Inbox currently shows all conversations unsorted before the JS sort anyway; pushing the sort to DB is strictly better.

### Finding 5 — Activity detail roster: ScrollView, no windowing

`ActivityDetailScreen.tsx:1071`: the full screen is a `<ScrollView>`. Inside it, `approvedParticipants` (line 172, `useMemo` over all approved join_requests) is mapped to avatar components (line 1299):
```tsx
approvedParticipants.map((r) => <ParticipantAvatar ... />)
```

This is not a `FlatList` — all participant avatar nodes are mounted at once, inside a parent `ScrollView`.

**Today:** Rally games max 6–12 players. Imperceptible.  
**Wall:** 200-member rally → 200 avatar nodes mounted simultaneously: all profile photo URLs loaded, all shadow layers computed, all re-render on any state change in `ActivityDetailScreen`.

**Mechanical fix:** replace the participants section with a `FlatList` (using `scrollEnabled={false}` inside the outer `ScrollView` requires a `nestedScrollEnabled` pattern) or a simple flat map limited to first 20 with a "see all" expand. The most pragmatic fix at Rally's current scale is to cap preview at 20 and add a "View all (N)" sheet — which the CLAUDE.md architecture already anticipates via `rosterItems` + `rosterTotal` props (line 1094–1095).
